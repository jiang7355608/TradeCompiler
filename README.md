# BTC 交易信号引擎

基于 OKX 15分钟K线的箱体突破交易信号系统，自动分析市场结构并生成做多/做空信号，通过邮件推送止损止盈点位。

## 策略思路

核心逻辑：**横盘蓄力 → 放量突破 → 趋势确认 → 延续验证**

### 市场结构分析

系统每15分钟收盘后自动拉取K线，分析以下结构：

1. **箱体识别**：最近N根K线的高低点价差是否在阈值内（判定横盘）
2. **双窗口突破**：当前收盘价必须同时突破短窗口（6根=1.5小时）和长窗口（14根=3.5小时）的高低点，防止小范围假突破
3. **放量验证**：当前K线成交量必须超过近20根均量的指定倍数
4. **趋势方向**：EMA5 vs EMA20 判断大趋势偏向，过滤逆势信号
5. **延续确认**：突破后价格必须继续朝突破方向运动（不能突破完就回头）

### 激进策略（Aggressive）

```
收到新K线
│
├─ 必须处于横盘蓄力状态
├─ 成交量放大
├─ 短窗口 + 长窗口突破方向一致
├─ 不逆势（趋势中性允许入场）
├─ 至少单根K线延续确认
├─ 过滤扫单K线（影线长实体小的蜡烛）
├─ 价格不在箱体中间（要在边缘突破）
├─ 同方向冷却期内不重复
│
└─ ✅ 发出信号
    ├─ 强延续（连续两根同向）→ 大仓位 + 高置信度
    └─ 普通延续（单根确认）→ 小仓位 + 低置信度
```

### 保守策略（Conservative）

与激进策略的区别：
- **不要求先横盘**（但有横盘时仓位加成）
- **趋势必须明确**，中性趋势直接拒绝
- **必须强延续**（连续两根K线同向），单根不够

信号更少，但入场质量理论上更高。

### 动态止损止盈

不使用固定比例，而是根据市场结构动态计算：

- **止损**：前一根K线的中点 `(high + low) / 2`
  - 逻辑：价格回到前一根K线中间，说明突破力度不够，认错出场
  - 兜底：止损距离不小于最小比例（防止止损太近被正常波动打掉）
- **止盈**：入场价 + 止损距离 × 风险回报比
  - 激进策略默认 3.0 倍，保守策略默认 1.5 倍

这样止损止盈会随市场波动自适应：箱体宽时止损宽止盈大，箱体窄时止损紧止盈小。

## 参数优化

内置网格搜索优化器，遍历关键参数组合并用历史数据回测，按综合评分排序：

```
score = 期望值 × √交易次数 × (1 - 最大回撤×2) × 利润因子
```

不是单纯选盈利最大的，而是综合考虑盈利质量、样本可信度和风险控制。

优化器只输出推荐参数，不会自动应用，由人工决定是否采纳。

## 技术栈

- Java 17 + Spring Boot 3
- OKX 公共 REST API（无需 API Key）
- Gmail SMTP 邮件通知
- 定时任务对齐K线收盘时间（cron `10 0/15 * * * *`）

## API 接口

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/signal/run` | 手动触发一次分析 |
| GET | `/api/strategy` | 查看当前策略 |
| POST | `/api/strategy/{name}` | 切换策略（aggressive / conservative） |
| POST | `/api/backtest/run` | 触发月度回测 |
| GET | `/api/optimize?top=5` | 参数网格搜索，返回 Top N 推荐参数 |

## 配置

所有参数在 `trading-signal/src/main/resources/application.yml`，修改后重启生效。

关键参数：
- `range-threshold`：横盘判定阈值，越小越严格
- `volume-spike-multiplier`：放量倍数门槛
- `agg-risk-reward-ratio` / `con-risk-reward-ratio`：风险回报比
- `agg-min-sl-pct` / `con-min-sl-pct`：最小止损兜底比例

## 运行

```bash
cd trading-signal
mvn spring-boot:run
```

回测：
```bash
mvn compile exec:java -Dexec.mainClass="com.trading.signal.backtest.BacktestRunner"
```
